# 语雀文档图片理解功能

## 概述

此功能允许在读取语雀文档时,自动识别文档中的图片并使用 GLM-4.6V 视觉模型理解图片内容,提供更全面的文档分析。

## 使用场景

### 适用场景
- 文档包含重要的图表、流程图、架构图
- 文档中的图片包含关键信息
- 需要完整理解文档内容（包括图片）
- 分析包含截图的技术文档

### 不适用场景
- 纯文本文档（无图片）
- 图片仅为装饰性质
- 用户仅关注文字内容

## 工作流程

### 1. 图片检测

读取文档后,自动检测以下格式的图片:

**Markdown 格式:**
```markdown
![图片说明](https://example.com/image.jpg)
```

**HTML 格式:**
```html
<img src="https://example.com/image.jpg" alt="图片说明">
<img src="https://example.com/image.jpg">
```

**常见语雀图片域名:**
- `cdn.nlark.com` - 语雀 CDN
- `mass.alipay.com` - 支付宝内网图床
- `gw.alipayobjects.com` - 支付宝对象存储
- `mdn.alipayobjects.com` - 支付宝 MDN 存储

### 2. 图片上下文提取

在询问用户前,先提取每张图片的上下文信息:

**提取内容:**
- 所在章节标题（向上查找最近的 # ## ### 标题）
- 图片前后的文字上下文（前后各 1-2 行）
- 图片的 alt 文本（如果有）
- 在文档中的顺序编号

**提取示例:**
```markdown
## 系统架构

我们的系统采用微服务架构：

![架构图](https://example.com/arch.png)

各服务之间通过 gRPC 通信...
```

提取结果:
- 所在章节: `系统架构`
- Alt 文本: `架构图`
- 上下文: `我们的系统采用微服务架构` / `各服务之间通过 gRPC 通信`

### 3. 用户确认（必须使用 AskUserQuestion 工具）

**重要：不要在文本中直接输出选项让用户手动回复，必须调用 AskUserQuestion 工具创建可点击的选项表单。**

**步骤 1：先输出图片位置信息**
```
文档中含有 X 张图片：
- 图片1：位于「章节名称」章节，描述XXX
- 图片2：位于「章节名称」章节，展示XXX
- 图片3：位于「章节名称」章节，显示XXX
```

**步骤 2：调用 AskUserQuestion 工具**
```json
{
  "questions": [{
    "question": "文档中含有X张图片，是否需要理解图片内容？",
    "header": "图片理解",
    "options": [
      {"label": "是，需要理解所有图片", "description": "使用视觉模型分析所有图片内容"},
      {"label": "是，只理解部分图片", "description": "选择性分析特定图片"},
      {"label": "否，只看文字内容", "description": "跳过图片，仅总结文字"}
    ],
    "multiSelect": false
  }]
}
```

**最佳实践:**
- 使用图片的 alt 文本来描述图片用途
- 如果没有 alt 文本,根据上下文推断图片内容
- 图片数量较多时(>5张),建议用户选择部分理解
- 用户选择"只理解部分图片"时,追问具体要理解哪些图片

### 4. 图片分析

对每张图片调用理解脚本:

```bash
python scripts/understand_image.py "<image_url>" "请详细描述这个图片的内容"
```

**可定制的提示词示例:**
- "请描述这个图片" - 通用描述
- "这是一个架构图,请说明主要组件和关系" - 架构图
- "这是一个流程图,请描述流程步骤" - 流程图
- "这是一个截图,请说明关键信息" - 截图
- "这是一个数据图表,请分析数据趋势" - 图表

### 5. 结果整合

将图片理解结果整合到文档总结中:

```
## 文档总结

### 文字内容
[文档文字部分的总结]

### 图片内容

**图片 1: [图片alt文本或位置]**
[GLM-4.6V 的理解结果]

**图片 2: [图片alt文本或位置]**
[GLM-4.6V 的理解结果]

...

### 综合分析
[结合文字和图片的整体理解]
```

## API 配置

### 配置文件位置

优先级顺序:
1. 当前工作目录: `.claude/settings.json`
2. 用户主目录: `~/.claude/settings.json`

### 配置格式

```json
{
  "env": {
    "ANTHROPIC_BASE_URL": "https://antchat.alipay.com/v1",
    "ANTHROPIC_AUTH_TOKEN": "your-api-key-here"
  }
}
```

**必要条件:**
- `ANTHROPIC_BASE_URL` 必须包含 `antchat.alipay.com`
- `ANTHROPIC_AUTH_TOKEN` 是有效的 API 密钥

### 错误处理

**配置错误:**
```
错误: 无法获取 API Key。请确保在 .claude/settings.json 或 ~/.claude/settings.json 中配置了 ANTHROPIC_AUTH_TOKEN,且 ANTHROPIC_BASE_URL 包含 antchat.alipay.com
```

**网络错误:**
```
错误: 请求失败 - [具体错误信息]
```

**API 错误:**
```
错误: API 返回格式异常 - [返回内容]
```

## 最佳实践

### 性能优化
- 图片数量较多时(>5张),提示用户处理可能需要一些时间
- 可以考虑批量处理或限制单次处理的图片数量
- 对于大型图片,API 请求可能较慢(设置 30 秒超时)

### 提示词优化
- 根据图片类型(架构图、流程图、截图等)定制提示词
- 使用更具体的问题获得更准确的理解结果
- 对于技术图表,明确要求说明关键技术点

### 结果呈现
- 按图片在文档中出现的顺序标注
- 保留图片的 alt 文本作为参考
- 将图片理解与前后文字内容关联
- 在总结中突出图片提供的关键信息

### 用户体验
- 始终先询问用户是否需要图片理解
- 明确告知图片数量和预计处理时间
- 失败时提供清晰的错误信息和解决方案
- 允许用户跳过图片理解直接查看文字内容

## 技术细节

### GLM-4.6V 模型

**模型特点:**
- 支持图片 URL 和 base64 输入
- 可以理解图表、截图、照片等多种图片类型
- 支持中英文提示和响应
- 提供详细的视觉理解结果

**请求格式:**
```json
{
  "model": "GLM-4.6V",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": "https://example.com/image.jpg"
        },
        {
          "type": "text",
          "text": "请描述这个图片"
        }
      ]
    }
  ],
  "stream": false
}
```

**响应格式:**
```json
{
  "choices": [
    {
      "message": {
        "content": "图片描述内容..."
      }
    }
  ]
}
```

### 脚本实现

`scripts/understand_image.py` 提供:
- 自动从配置文件读取 API Key
- 发送图片理解请求
- 解析和返回理解结果
- 完整的错误处理

**命令行用法:**
```bash
# 基本用法
python scripts/understand_image.py "https://example.com/image.jpg"

# 自定义提示词
python scripts/understand_image.py "https://example.com/image.jpg" "这是一个架构图,请说明主要组件"
```

## 示例

### 示例 1: 技术文档中的架构图

**文档内容:**
```markdown
# 系统架构

我们的系统采用微服务架构:

![系统架构图](https://example.com/architecture.png)

各服务之间通过 gRPC 通信...
```

**图片理解结果:**
```
这是一个系统架构图,展示了微服务架构的整体布局。图中包含:
1. 前端层: Web 应用和移动应用
2. API 网关: 统一入口,负责路由和认证
3. 服务层: 包括用户服务、订单服务、支付服务等多个微服务
4. 数据层: 每个服务有独立的数据库
5. 基础设施层: 包括服务注册中心、配置中心、消息队列等
各组件之间通过箭头标注了通信方式和数据流向。
```

### 示例 2: 流程图

**文档内容:**
```markdown
# 审批流程

![审批流程](https://example.com/workflow.png)
```

**定制提示词:**
```bash
python scripts/understand_image.py "https://example.com/workflow.png" "这是一个审批流程图,请按顺序说明各个步骤"
```

**图片理解结果:**
```
这是一个审批流程图,包含以下步骤:
1. 开始: 用户提交审批申请
2. 初审: 直属主管进行初步审核
3. 判断节点: 金额是否超过 1 万元
   - 是: 进入财务审批
   - 否: 直接进入最终审批
4. 财务审批: 财务部门审核预算和资金
5. 最终审批: 部门负责人最终批准
6. 结束: 审批完成,通知申请人
流程图用不同颜色标注了不同审批人的职责范围。
```

## 限制和注意事项

### API 限制
- 需要有效的 API Key
- 请求超时设置为 30 秒
- 需要稳定的网络连接

### 图片要求
- 图片 URL 必须可公开访问（或在授权范围内）
- 支持常见图片格式（JPG、PNG、GIF 等）
- 图片不宜过大（建议 < 5MB）

### 理解质量
- 模型理解质量取决于图片清晰度
- 复杂的专业图表可能需要更具体的提示词
- OCR 文字识别准确度可能受图片质量影响

### 隐私和安全
- 图片 URL 会发送到 antchat.alipay.com
- 确保图片不包含敏感信息
- 遵守公司数据安全政策
